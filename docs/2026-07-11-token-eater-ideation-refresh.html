token-eater's next chapter — the harvester that shipped

Refreshed 2026-07-11 Originally 2026-06-26 Topic token-eater (was surplus-credit-harvester) Mode repo-grounded · ~/SITES/token-eater @ v0.1.1

The original run (below, preserved as history) imagined a piece of standing personal infrastructure: one idempotent loop harvesting about-to-expire credits across every model CLI. That ideation inspired a real project — and the project has since shipped and diverged from it three times over. token-eater is now a distributable House of Vibe skill (v0.1.1): a member types /token-eater, picks a service, and that service runs a whole cleanup skill on its own credits, in an isolated worktree, landing a draft PR token-eater never merges. This refresh re-grounds against what it became, records what happened to the original six ideas, and ranks the strongest next moves given where the product actually is — with its flagship promise (spend visibility) currently broken and its biggest safety and distribution work still open.

This is a resume of the 2026-06-26 doc, not a fresh sheet. The original grounding, the six ideas it produced, and its rejection table are carried forward — §02 is an honest ledger of what became of each. The new grounding (§01), refreshed axes (§03), ranked ideas (§04), and cuts (§05) reflect the built reality: a triple pivot from fleet scheduler → meta-skill, from mechanical-chore delegation → recipe-driven whole-skill delegation, and from personal infra → shippable member product.

01Grounding Context — what token-eater became

The meta-skill, in one breath

You point token-eater at one service (grok/codex/claude) and a token-heavy maintenance skill. That service runs the whole job on its own credits — it loads your installed skills (de-monolithize, de-slopify, ce-*), runs the skill, keeps the gate green, reviews-and-fixes its own work, opens a draft PR. token-eater is the launcher + guardrail: clean worktree off origin/main, hand over one recipe, then independently re-run the gate and make sure nothing merges.

Registry-driven: adapters.yaml says it plainly — "There is no posture, tier, or cost-routing engine: you choose the service, and the run loop spends it on any chore the project's own checks can verify."

The triple pivot from the original doc

  • Fleet scheduler → meta-skill. PR #12 deliberately collapsed the cross-provider drain loop. Posture / reserve floor / cost-routing were removed, not deferred lightly.
  • Chore delegation → recipe delegation. The crown-jewel insight: the service is a full agent that already runs your skills — token-eater doesn't re-implement work, it hands over a recipe.
  • Personal infra → HoV member product. A plugin (plugin.json, v0.1.1). A non-technical member types /token-eater, answers a plain-language preflight once, and never sees a flag, gate, or skill name again.

Verification, softened but honest

The original strict rule ("deterministic gate or keep on Claude") became a 3-tier confidence ladder: A "tests pass" (test&&typecheck), B "structural checks pass" (typecheck/build/lint), C soft — no gate, so it still runs but the draft PR is loudly labeled "AI-reviewed only — please read before merging."

token-eater re-runs the gate itself after the service finishes (self-report never trusted). Review is service-aware: claude→ real /ce-code-review; grok→ persona-file fleet. Layered review (ce-code-review → pro-gate) was proven on token-eater's own code — each tier caught bugs the others missed.

The new open problems the original never saw

  • Spend is invisible on grok (issue #9). The whole pitch is "burn your credits," but grok's headless log emits one usage record (the orchestrator turn); the subagents doing the real work log nothing. The flagship promise is unverifiable for the flagship service.
  • Distribution doesn't exist yet. The HoV app has no skill-install mechanism — bundle format, versioning, the ce-* plugin dependency all unresolved.
  • The service runs unsandboxed (issue #29). pro-gate twice flagged an OS sandbox as required before untrusted-repo distribution.
  • Still in beta (issue #3): disable-model-invocation:true — a human must type it.

02What became of the original six

The 2026-06-26 ideas were the Grok adapter's spec. Two weeks and ~16 merged PRs later, most shipped, one was deliberately dropped, and the grand one was simplified away. This is the honest ledger — it is why the new ideas below start where they do.

#Original ideaWhat actually happened
1 Surplus-credit harvester — one scheduler, every provider simplified away PR #12 collapsed the fleet scheduler to a meta-skill: one service per run, no posture/reserve/cost-routing. The adapter seam survived (adapters.yaml). The scheduler is now a deferred "power-user phase 2" — cross-repo sweep (#5) and reserve-floor oracle (#4) are its open remnants.
2 Deterministic-gate eligibility + Grok-safe taxonomy shipped, softened Became the 3-tier gate — Tier C now allows gateless runs (labeled) for accessibility, an explicit reversal of "gate or nothing." The taxonomy became skills-catalog.yaml and the plain-language member menu.
3 Clone the Codex harness — worktree, structured, PR-not-merge shipped + Built (run-session.sh, wt.sh, delegate-*.sh). Plus safety the original never listed: never touch the member's checkout (#26), trust gate, repo-string validation, .env not copied (#16), wall-clock runaway backstop (#28).
4 Headless-contract preflight + scope-fenced prompt shipped + Built (check-auth.sh, detect-adapters.sh). Plus a whole dimension it missed: macOS stock bash 3.2 + BSD-tools portability (#14) is now a standing pre-distribution checklist item.
5 Per-codebase eval harness + Grok trust ledger half-built The run ledger exists (.token-eater/runs/ledger.jsonl) and independent re-verify + layered review shipped — but the ledger is write-only. The auto-demoting trust ledger / eval calibration was never built. This is the seed of new idea #3.
6 Grok-chores backlog queue (beads-backed) dropped Replaced by auto-discovery: the skill finds its own target on the service's credits (SKILL.md step 4). The curated queue was judged unnecessary for the member flow.

04Ranked Ideas — the next chapter

Top pick

#1 — answer "what did this cost me," honestly. token-eater's headline is "burn your credits," yet for grok, its flagship drain target, the spend line is empty. Fixing this is not a metrics nicety — it is closing the one gap between the product's promise and what a member can actually see. Five independent ideation frames converged here, and the fix is grounded: the ledger already has a spend_estimate slot waiting to be filled. Its natural partners are #2 (the sandbox that unblocks distribution) and #3 (reading the ledger back so the tool finally compounds).

1

Answer "what did this cost me," honestly

A4 · Economics & observability Confidence 85% Complexity Medium Top pick · 5-frame convergence
Description

Stop trying to log a number grok will never emit, and give the member a truthful answer three ways, best-available-first. (a) Weigh the wallet, don't read the receipt: snapshot the account balance immediately before and after the run and report spend = before − after (issue #9's GetGrokCredits reader, made self-contained — no onwatch dependency, since members won't run that daemon). (b) When no balance exists, estimate and label it — a FinOps-style same-run estimate calibrated against claude/codex's real cost_usd on the same archetype, shown as "≈ comparable to $X," never a blank line, trued-up later if a reader lands. (c) Lead with the number the member actually acts on: a "review receipt" (files × diff size × confidence tier → "5-minute skim / 20-minute read / get a second opinion first"), which every service can compute today. The dollar figure is secondary; the review-time is primary.

Basis

direct: DISTRIBUTION-READINESS.md — "Per-run spend is unobservable on grok … the parallel native subagents that do the real work log no token usage … the DONE-block spend line is empty for grok" while "claude/codex delegates DO report cost_usd." result-handling.md already carries a spend_estimate field (currently "unknown") and an event:"update" enrichment path for a later true-up. external: FinOps committed-use burndown reconciles a labeled same-day estimate against the true invoice later; balance-delta metering is how prepaid systems measure consumption when itemized receipts are unavailable.

Rationale

This is the smallest change that makes the product's core claim true. Issue #9's current framing bets everything on one brittle gRPC integration; the three-tier answer ships value now (the estimate and the review receipt need no new API) and folds the reader in as an upgrade, not a blocker. And reframing the "cost" as review attention answers the question the member is actually asking.

Downsides & the hard part

The balance-delta is only clean if no other grok activity overlaps the run window — concurrent use poisons the subtraction, so it needs a "measured during exclusive window" caveat. The estimate's calibration drifts as model pricing changes and must be re-fit. And a wrong-but-confident dollar figure is worse than an honest "estimated" label — the labeling discipline is load-bearing.

2

Graduated OS containment, shipped with distribution

A2 · Trust & safety Confidence 88% Complexity High Named distribution blocker
Description

--trust-repo is a consent checkbox, not a technical control: once granted, the service (grok --always-approve, claude --allowedTools Bash) and the gate run an unsandboxed shell under the member's OS account — a booby-trapped repo or a model mistake can reach $HOME, credentials, and the network. Do two things. Make the sandbox the default, not a later bolt-on: run every session inside sandbox-exec (macOS) / bwrap (Linux), mirroring codex's workspace-write posture that already models the target, with an explicit named opt-out for the rare gate that needs broad access. Grade containment by repo-risk, biosafety-style: derive a tier from concrete signals (first-run vs Nth-clean-run, --install-deps requested, the member's own repo vs a fresh clone) and let the ledger earn relaxation over clean history — trust becomes evidence-based and reversible, not a permanent yes. Tie the whole thing to a hard rule: the sandbox ships in the same release as the install mechanism — distribution multiplies the blast radius, so they cannot ship apart.

Basis

direct: DISTRIBUTION-READINESS.md — "an OS sandbox around the service + gate is the required hardening before distribution beyond trusted-repo use, and codex (workspace-write) already shows the target posture"; the pro-gate verify-round re-flagged it 2026-07-03; issue #29 pushes even to an OS-level resource fence. external: BSL 1–4 biosafety containment assigns level by agent risk and enforces it with engineering controls, not a signed form — the direct analogue to "consent click" vs "kernel boundary."

Rationale

This is the single named item standing between "internal-ready" and "safe for a non-technical member on a repo they cloned from a tutorial." Every other safety item in the readiness doc is done. Shipping the install mechanism first — as currently sequenced — is shipping issue #29 at 10,000× the reach; the biosafety grading is what keeps the safe path fast for repeat users instead of taxing every run forever.

Downsides

A default sandbox breaks legitimate gates that need filesystem or network (the exact reason the trust gate shipped first) — the opt-out and per-ecosystem allowances have to be right or Tier A/B silently degrade. Cross-platform sandboxing (sandbox-exec vs bwrap vs Windows/WSL) is real, bash-3.2-constrained work. And "earned relaxation" needs the ledger read-back from #3 to exist.

3

Close the compounding loop — let merges set the tier

A6 · Compounding A3 · Verification Confidence 82% Complexity Medium
Description

token-eater writes a rich ledger.jsonl after every run — provider, gate_outcome, result_ref (the PR URL), files_modified — and then never reads it back. A tool built by a "compound engineering" practice that doesn't compound its own delegation history is the most on-the-nose gap in the project. Close it: mine the ledger (plus each PR's actual disposition via gh pr view — merged untouched? heavily edited? closed?) to bin every (service, skill, repo) triple by real track record, semiconductor-style, and route three things off that bin — (1) the preflight's "(Recommended)" badge; (2) review depth: a combo with a clean five-run history can fast-track, one that keeps getting rewritten auto-escalates to pro-gate before the PR even opens; (3) discovery: skip a chore already PR'd or gate-failed within N days. The same read-back subsumes issue #10's duplicate-PR problem — the ledger becomes a cross-session claim record ("archetype X on repo Y in flight, run Z"), reusing the flock that already exists, without resurrecting the scheduler PR #12 removed.

Basis

direct: result-handling.md defines the append-only ledger.jsonl with exactly these fields and an event:"update" path, yet no reference reads it; grounding confirms "the auto-demoting trust ledger / per-codebase eval harness NOT built." worktree-lifecycle.md — "two sessions could independently pick the same chore and open duplicate PRs … a chore-claim file is the planned fix" — and the per-repo flock already exists one hop away. external: semiconductor post-fab binning grades rather than scraps, and scales inspection sampling to recent defect history.

Rationale

Every other axis leans on this. #1's true-up wants ledger enrichment; #2's earned sandbox relaxation wants clean-run history; #5's escalation wants a defect signal. Building the read-back once turns the data token-eater already throws away into the engine that makes confidence, safety, and cost all compound — the exact promise the "eater" name has been writing IOUs against since the ledger shipped.

Downsides

Auto-demotion on a tiny sample thrashes — needs a minimum-N floor before a bin influences routing. "Merged untouched" is a noisy proxy for "correct" (a member may merge blind), so PR disposition informs but cannot be the sole signal. And the ledger is per-repo local (.token-eater/, uncommitted), so history evaporates on a fresh clone until #6/distribution gives it somewhere durable to live.

4

Recipe chains — a whole spring-cleaning pass, not six invocations

A1 · Meta-skill core Confidence 75% Complexity Medium
Description

The session boundary is drawn at "one service, one skill, one PR" — but skills-catalog.yaml holds six independently-gated archetypes that are naturally complementary (de-monolithize a fat module, then remove the dead code the split exposed, then tidy formatting). Redraw the boundary at the worktree, not the archetype: let one session run a short ordered chain of 2–3 archetypes in the same worktree, each gated independently before the next begins, closing with one combined draft PR and one combined confidence label — the lowest tier in the chain wins, so honesty is preserved. This turns "spend my grok credits on de-monolithize" into "give my repo a spring-cleaning pass" — which is what a non-technical member actually wants, and it re-uses the expensive worktree + dependency-install setup across all three steps instead of paying it three times.

Basis

direct: SKILL.md — "A token-eater run is one session: one service, one skill, one polished draft PR" — set against skills-catalog.yaml, whose six model-exec archetypes each carry their own gate and are sequentially compatible (dead-code's gate is "removed symbols have no references"; de-monolithize's is "full suite passes unchanged" — composable in that order). reasoned: the per-session cost (worktree, dep install, preflight, review dispatch) is fixed overhead amortized better across a chain; the only new logic is sequencing + a lowest-tier-wins combiner.

Rationale

It reframes token-eater from "a chore runner" to "a cleanup crew" — the difference a member feels. It also spends surplus credits more completely in one drain (the original doc's whole point), and it stays inside the shipped safety model: each step still gates independently, still lands as a draft PR, still never merges.

Downsides

A mid-chain gate failure needs a clean policy — ship the steps that passed, or roll the whole chain? A combined PR is harder to review than three focused ones (partly why the current boundary exists), so the PR body must clearly segment "what each step did." And chains multiply run time and blast radius, which raises the stakes on #2's sandbox.

5

Finish the review step you already tell the human to finish

A3 · Verification Confidence 78% Complexity Medium
Description

Two gaps sit at the review step, both aimed at the exact non-technical member the rest of the design protects. First: SKILL.md's final instruction tells the human to go do more work — "for your own peace of mind, run a fresh /ce-code-review … before merging" — a manual step that only happens if they remember the sentence. Fold it in: run that independent cross-model pass (pro-gate or a second-model review) as the actual last step of run-session.sh for Tier A/B, before the PR is shown, so "the service reviewed its own work" is never the final state. Second: the moment the PR opens, that same member is handed a raw unified diff and a slash command — the single highest-jargon moment in the whole flow. Replace it with the primitive the onboarding already uses: walk them through the top N hunks conversationally (AskUserQuestion: keep / revert / explain), and the PR takes its final shape only after that dialogue.

Basis

direct: SKILL.md step 6 hands the member an independent-review homework assignment and a raw diff — directly contradicting its own Audience-first framing that the member "does NOT know what a gate, a flag, a parameter … is." DISTRIBUTION-READINESS.md proves the extra pass earns its keep: "layered review … each tier caught complementary bugs the others missed." reasoned: a 3-tier gate and independent re-verify are worthless as member-facing safety if the member's actual last check is "read a diff" the product says they can't read.

Rationale

token-eater's whole trust story is "we prove it, honestly." Right now the proof has a hole where the strongest check is optional-if-remembered and the member-facing check regresses to jargon. Closing both makes the safety real for the audience it's built for — and the conversational review doubles as the plainest possible answer to #1's "how carefully do I need to read this?"

Downsides

An always-on pro-gate pass adds latency and (on paid tiers) cost to every run — it may belong behind a "high-stakes only" trigger keyed to #3's bin. Conversational hunk review is a real UX build, and for a large diff "top N hunks" risks lulling the member past the ones that matter — hunk selection has to be risk-ranked, not size-ranked.

6

Escape beta by nudging idle credits, not flipping the injection surface

A5 · Distribution & UX Confidence 72% Complexity Medium
Description

Issue #3 ("promote out of disable-model-invocation beta") reads like a binary switch — but flipping it globally lets any installed skill hand off to token-eater's unsandboxed, credit-burning, PR-opening shell, reopening the exact injection surface issue #29 is trying to close. The real gap is narrower: a member with idle, about-to-expire credits who simply doesn't remember to type /token-eater. Solve that instead. Surface the already-shipped schedule-install.md path as an opt-in idle-credit nudge — a cron/Stop-hook trigger that still asks ("your grok credits reset in 2 days and you've got a backlog — run a cleanup?"), never silently executing — and gate any true autonomy on per-repo earned trust (a repo already --trust-repo'd with a clean ledger history from #3), not a global flag flip.

Basis

direct: issue #3 and references/schedule-install.md (the recurring-schedule capability already exists, cited in SKILL.md's reference map); issue #29 and the unsandboxed-shell admission are why a naive flag flip is unsafe. reasoned: the obvious fix ("disable-model-invocation:false") maximizes new attack surface to solve a "member forgot" problem; a nudge gets ~90% of the value (token-eater runs without the member remembering) with none of the arbitrary-skill-handoff exposure.

Rationale

Adoption, not features, decides whether this product matters — and adoption dies on "the member forgot the tool exists." The nudge is the honest bridge out of beta: it makes idle credits get spent without handing the keys to every skill on the machine, and it composes with #1 (the nudge can quote "you left ≈$X of grok on the table last week").

Downsides

A credit nudge is an outbound interruption — get the frequency wrong and it's spam a member disables. It needs a real "credits about to expire" signal, which circles back to #1's observability gap (grok's is the weakest). And a background trigger on an unsandboxed tool still wants #2 shipped first before it runs anything without a human present.

7

Extract the shared delegation core — before three more things fork it

A1 · Meta-skill core A2 · Safety Confidence 85% Complexity Low–Medium
Description

Safety-critical behavior — trust-gate enforcement, .env scrubbing, repo-string validation, the wall-clock runaway backstop, ledger writes — currently lives duplicated across delegate-grok.sh, delegate-codex.sh, and delegate-claude.sh. Do the issue #7 extraction into one audited delegation library now, because three big pending items each carry an implicit "× N delegate scripts" tax: the sandbox fence (#29/#2), the balance reader (#9/#1), and the cursor/gemini adapters (#6). Land the shared core first and each becomes a 1× change instead of a 3× hand-threaded patch that can silently miss one adapter.

Basis

direct: open issue #7 ("extract shared delegation library from delegate-*.sh"); commit c3432f4 ("working claude delegate, bash-3.2 grok delegate") shows safety fixes landing per-adapter, one at a time, in practice. The live proof: worktree-lifecycle.md still documents copying .env* while DISTRIBUTION-READINESS says .env copying was removed — an invariant that already drifted between two places. reasoned: a safety property is only as strong as its weakest re-implementation; three copies is three chances to regress one while "fixing" another.

Rationale

The least glamorous idea here and the highest-leverage sequencing move. It's not a feature — it's the thing that makes the next three features cheap and the safety story trustworthy. A strong team looks at "when the sandbox lands, does it touch one file or three?" and does this first.

Downsides

Pure refactor with no member-visible payoff, so it's easy to skip under feature pressure — exactly when the drift bites. The three adapters have genuinely different exec shapes (grok's soft schema, codex's workspace-write, claude's permission modes), so the shared core must abstract without flattening those real differences. And it must stay bash-3.2-clean.
